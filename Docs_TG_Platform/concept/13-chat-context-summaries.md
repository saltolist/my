# Механика сводок контекста чата

Этот документ описывает, как TG Platform передаёт в ИИ контекст канала, историю диалога и технические сводки, а также как это должно отображаться в лог-чате платформы.

---

## Зачем нужна механика

У ИИ есть ограниченное контекстное окно. При этом платформа должна учитывать:

- профиль канала: тема, аудитория, тон, рубрики, правила, запреты;
- последние сообщения текущего чата;
- старую часть диалога, которая уже не помещается в последние сообщения;
- дополнительные данные запроса: найденные заметки, web search, контекст поста.

Чтобы не передавать весь профиль канала и всю историю в каждом обычном пользовательском сообщении, платформа использует отдельные служебные пары сообщений.

---

## Термины

**Сводка канала** — компактный снимок `channel_context`: профиль канала, тон, рубрики, правила, запреты и другие настройки, которые влияют на ответы ИИ.

**Саммари диалога** — краткое резюме сообщений, которые вышли за лимит последних 8 пар `user/ai`.

**Синтетический ответ** — невидимый для пользователя ответ ИИ `Понял, учту.`, который подтверждает, что служебная сводка принята в контекст.

**Лог-чат** — техническое зеркало обычного чата. В нём видны скрытые сообщения, системный prompt, сводки, summary и дополнительные данные, которые не показываются в обычном чате.

---

## Создание нового чата

Когда пользователь отправляет первое сообщение и этим создаёт новый чат, реальный контекст для ИИ строится не с пользовательского запроса, а с невидимой служебной пары.

Порядок сообщений в лог-чате:

1. Скрытое сообщение платформы:
   - `channel_context` — сводка канала на момент создания чата;
   - `dialog_summary` — саммари старой части диалога. Для нового чата обычно пустое: «Сообщений за пределами последних 8 пар пока нет».
2. Синтетический ответ ИИ:
   - `Понял, учту.`
3. Первое видимое сообщение пользователя:
   - сам текст запроса;
   - `system_prompt`;
   - данные конкретного запроса: найденные заметки, web search, контекст поста;
   - последние 8 пар диалога, если они уже есть.
4. Ответ ИИ.

В обычном чате пользователь видит только пункты 3 и 4. Пункты 1 и 2 видны только в лог-чате.

---

## Что передаётся с обычным сообщением пользователя

Обычное сообщение пользователя не должно каждый раз нести сводку канала.

С ним передаются только данные, относящиеся к конкретному запросу:

- `system_prompt`;
- `last_8_pairs` — последние 8 пар `user/ai` текущего чата;
- `retrieved_notes` — найденные релевантные заметки;
- `web_results` — найденная в интернете информация, если включён web search;
- `post_context` — контекст поста, если чат локальный внутри поста;
- настройки маршрута модели: LLM, Web Search, multi-response.

`channel_context` уже присутствует в первой служебной паре чата и не дублируется в каждом пользовательском сообщении.

---

## Саммари старой истории

ИИ обычно получает последние 8 пар сообщений. Всё, что вышло за этот лимит, сжимается в `dialog_summary`.

`dialog_summary` живёт в служебной первой паре вместе со сводкой канала. Оно нужно, чтобы ИИ понимал старые решения, договорённости и контекст, не держа в prompt весь диалог.

При росте чата summary должно обновляться серверной логикой:

- последние 8 пар остаются в явном виде;
- более старая часть диалога сжимается;
- новая версия summary используется в служебной паре контекста.

В демо это показывается как mock-механика в лог-чате.

---

## Изменение контекста канала

Если пользователь меняет профиль канала, старая сводка в уже существующих чатах не заменяется мгновенно.

У каждого чата есть версия сводки канала:

- при создании чата фиксируется текущая версия `channel_context`;
- если профиль канала изменился, глобальная версия сводки становится новее;
- существующий чат остаётся на старой версии, пока пользователь не отправит следующее сообщение.

Когда пользователь отправляет следующее сообщение в чат со старой версией сводки, вместе с этим сообщением передаётся `channel_context_update`.

Это обновление содержит:

- старую и новую версии сводки;
- новую сводку канала;
- summary диалога, если часть истории уже вышла за лимит последних 8 пар.

После обработки этого сообщения новая сводка «догоняет» чат и становится его базовой сводкой. В лог-чате это выглядит как отдельный скрытый блок при соответствующем пользовательском сообщении.

---

## Как это выглядит в лог-чате

Лог-чат должен показывать техническую реальность, а не только обычную UI-историю.

В нём видно:

- невидимую первую пару `channel_context + dialog_summary → Понял, учту.`;
- все видимые сообщения обычного чата;
- все ветки пользовательских сообщений после редактирования;
- для каждого пользовательского запроса — что именно увидит ИИ;
- для AI-ответов — модель, web search, multi-response варианты и размер контекстного окна;
- обновления `channel_context_update`, если профиль канала изменился и сводка догоняет чат.

Лог-чат не является отдельным пользовательским чатом. Это техническая проекция существующего чата.

---

## Важный инвариант

Сводка канала не передаётся в каждом пользовательском сообщении.

Она передаётся:

1. один раз при создании чата — как первая невидимая служебная пара;
2. повторно только при изменении контекста канала — как `channel_context_update` в следующем пользовательском запросе, после чего становится новой базовой сводкой чата.

Все остальные пользовательские сообщения несут только данные конкретного запроса и последние 8 пар диалога.
