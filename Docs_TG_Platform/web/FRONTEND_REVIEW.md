# Ревью фронтенда TG Platform (`web/`)

> Цель ревью: оценить готовность фронтенда к подключению бэкенда «без правок фронтенда»,
> качество кода, архитектуру, актуальность подхода и технический долг.
> Документация в оценку **не входила** (будет переписана).
>
> Дата: 16 июня 2026. Объём: `src/` — 528 файлов `.ts/.tsx`, 35 тестовых файлов, 171 тест.

| Чистота кода | **9.5 / 10** | 0 `any` в TS, 0 `TODO/FIXME`, 0 `@ts-ignore` |
| Современность стека | **10 / 10** | Next 16, React 19, TS strict, TanStack v5, Zod v4 |

| Технический долг | **8 / 10** | Долг точечный и сконцентрированный, не размазан |

**Вердикт.** Это очень сильный, зрелый фронтенд — один из самых чистых кодовых
баз такого размера: строгая FSD-слоистость с автоматическим контролем границ,
паттерн Repository, TanStack Query с оптимистичными апдейтами, аккуратная
декомпозиция (нет «god-файлов» — крупнейший модуль кода ~394 строки).

Но заявленная цель — **«подключить бэкенд, не трогая фронтенд»** — пока
**не достигается полностью**. Repository-граница закрывает CRUD, однако
4 элемента контракта данных «вшиты» во фронтенд так, что реальный бэкенд
потребует правок. Они перечислены в разделе 3 и являются главным фокусом доработки.

## 2. Что сделано отлично (сохранить как есть)

- **FSD с реальным энфорсментом.** Слои `app → screens → widgets → features → entities → shared`
  контролируются `eslint-plugin-boundaries` (`eslint.config.mjs`), импорты только вниз,
  публичный API через `index.ts`. Это не «договорённость на словах», а проверяемое правило.
- **Repository pattern + переключение источников.** `createRepositories.ts` выбирает реализацию
  (`http` / `msw` / `seed`) по `dataSource.ts`. Интерфейсы (`repositories.ts`) отделены от
  реализаций (`httpRepositories.ts`, `seedRepositories.ts`). Переключение на бэкенд — это
  две env-переменные. Это правильный фундамент.
- **MSW как контракт.** `msw/handlers.ts` точно повторяет REST-эндпоинты HTTP-репозиториев —
  фронтенд разрабатывается против настоящих сетевых вызовов, а не моков в памяти.
- **Server-state дисциплина.** `queryKeys.ts` — иерархические ключи со скоупом по аккаунту
  (`["account", accountId, ...]`), что даёт корректную инвалидацию и изоляцию данных между
  пользователями. Оптимистичные апдейты с откатом (`usePosts.ts`).
- **Чистота кода уровня «эталон».** Ноль `any` в TS-коде, ноль `TODO/FIXME/HACK`, ноль
  `@ts-ignore`. Единичные `eslint-disable` — обоснованные (`next/no-img-element`).
- **Container/Presenter в экранах.** Модель-хук возвращает структурированный
  `{ data, ui, actions }` (см. `useFeedScreen.tsx`) — UI-компоненты остаются «глупыми».
- **`tsconfig` strict, современные таргеты** (`ES2022`, `bundler`), алиасы по слоям.
- **Auth-граница изолирована** в `entities/auth/api/authApi.ts` + `shared/lib/auth/session.ts`.

## 3. Критично для подключения бэкенда (главный фокус)

Эти 4 пункта — причина, по которой цель «не трогая фронтенд» сейчас не выполняется.
Все они нарушают принцип «фронтенд владеет своим состоянием, бэкенд владеет контрактом данных».

### 3.1. Симуляция AI-ответов живёт в app/feature-слое, а не за репозиторием 🔴

Главный «интеллект» платформы (ответы ассистента) синтезируется прямо в UI-логике
через `setTimeout` и хардкод-словарь, **минуя** Repository-границу:

- `app/model/store/composer-store.tsx` — `window.setTimeout(900)` → `buildAiReplyMessage(...)`
- `shared/lib/replies.ts` — `getGlobalReply` / `getPostReply` (захардкоженные тексты)
- зеркально продублировано в `seedRepositories.ts` и `msw/handlers.ts`

```111:131:Docs_TG_Platform/web/src/app/model/store/composer-store.tsx
      const id = `gc${Date.now()}`;
      const newChat: GlobalChat = { id, title: truncate(text, 40), preview: text, date: "сейчас", history: [{ role: "user", text }] };
      void createChat.mutateAsync(newChat).then(() => {
        bridge.goToHref(routes.gchat(id));
        window.setTimeout(() => {
          const reply = buildAiReplyMessage(cfg, getGlobalReply(text), "home", getTarget("home"));
          void pushMessage.mutateAsync({ chatId: id, message: reply });
        }, 900);
      });
```

**Почему это блокер.** Когда придёт реальный AI-бэкенд, ответ должен приходить с сервера
(`POST /messages` → ответ ассистента в теле/через стрим), а не генерироваться на клиенте.
Чтобы это включить, придётся **переписать `composer-store`** и виджеты `chat-thread` —
то есть «трогать фронтенд».

**Рекомендация.** Ввести в `RepositoryBundle` отдельный контракт, например
`AssistantRepository.sendMessage(scope, chatId, text): Promise<ChatMessage>` (или стрим).
Композер должен только вызывать репозиторий и рендерить ответ; `seed`/`msw`-реализации
оставляют текущую заглушку с задержкой, а `http`-реализация ходит на бэкенд. После этого
подключение AI = смена реализации репозитория, без правок UI.

### 3.2. Даты хранятся как человекочитаемые строки, а не ISO-8601 🔴

В модели данных `date` — это строка вида `"28 апр · 14:22"`, `"сейчас"`, `"только что"`
(см. `shared/data/seed-data.ts`). Есть даже обратный парсер по русским словам:

```72:94:Docs_TG_Platform/web/src/shared/lib/helpers.ts
/** Парсит строку даты поста: «28 апр · 14:22». */
export function parsePostDateTime(raw: string | undefined | null): Date | null {
  ...
  const month = RU_MONTHS_3[m[2].slice(0, 3)];
  ...
}
export function postFreshness(post: Post): number {
  const raw = (post.date || post.created || "").trim().toLowerCase();
  if (/^(только что|сейчас|сегодня)/.test(raw)) return Date.now();
  return parsePostDateTime(raw)?.getTime() ?? 0;
}
```

**Почему это блокер.** Реальный бэкенд вернёт ISO-таймстемпы (`2026-04-28T14:22:00Z`).
Весь слой отображения (`Post.date`, `GlobalChat.date`, сортировка ленты, группировка по дням)
завязан на предварительно отформатированную русскую строку. Бэкенд не сможет отдавать такой
формат — придётся переделывать модель и парсер.

**Рекомендация.** Хранить во всех сущностях ISO-8601 (UTC). Форматирование в
`"28 апр · 14:22"` / `"Сегодня"` / «сейчас» вынести в чистые функции отображения
(`formatPostDate(iso, locale)`), вызываемые в UI. Тогда `seed`/`http` отдают одинаковый
машинный формат, а локализация — забота презентационного слоя.

### 3.3. ID генерируются на клиенте; нет реконсиляции с серверным ID 🟠

Идентификаторы создаются через `Date.now()` (`composer-store.tsx`, `useNoteEditor.tsx`,
`filterPosts.ts`, модели профиля). При этом тип ID неконсистентен: `Post.id: number`,
а `GlobalChat.id: string` / `GlobalNote.id: string`.

В `useCreatePost` оптимистично кладётся объект с клиентским ID, а в `onSuccess`
реконсиляции с серверным ID нет (MSW просто эхо-возвращает тот же объект):

```52:69:Docs_TG_Platform/web/src/entities/post/model/usePosts.ts
  return useMutation({
    mutationFn: (post: Post) => posts.create(post),
    onMutate: async (post) => { ... queryClient.setQueryData(queryKeys.posts.detail(accountId, post.id), post); ... },
    onError: (_error, _post, context) => { ... },
  });
```

**Почему это важно.** Настоящий бэкенд присвоит **свой** ID. Тогда оптимистичная запись
в кэше (`detail(accountId, clientId)`) останется «осиротевшей», а ссылки/URL вида
`routes.post(clientId)` будут указывать в никуда.

**Рекомендация.** (1) Унифицировать тип ID (строковый `string`/UUID предпочтительнее для всех
сущностей). (2) В `onSuccess` мутаций `create` заменять временную запись на серверную сущность
(сдвиг ключа `detail` с клиентского ID на серверный, обновление `list`). (3) Клиентский ID
трактовать как временный «tempId» только до ответа сервера.

### 3.4. Ответы бэкенда не валидируются Zod-схемами 🟠

Схемы (`shared/api/schemas/*`) описаны качественно (`postSchema`, `globalChatSchema`, …),
но в боевом пути данных **не применяются**: `httpRepositories.ts` приводит ответ кастом
`apiRequest<Post[]>`, а сам `httpClient.ts` делает `JSON.parse(text) as T` без `.parse()`:

```60:64:Docs_TG_Platform/web/src/shared/api/httpClient.ts
  if (res.status === 204) return undefined as T;
  const text = await res.text();
  if (!text) return undefined as T;
  return JSON.parse(text) as T;
```

Сейчас Zod-схемы используются только в тестах. Параллельно существуют **две** независимые
системы типов: ручные типы в `shared/types/index.ts` (`Post`) и Zod-инференс (`PostDto`) —
риск рассинхронизации.

**Почему это важно.** При подключении настоящего API любое расхождение формата (новое поле,
`null` вместо строки, пропуск поля) проникнет в рантайм незаметно и «взорвётся» далеко от
источника. Это именно тот класс багов, который должен ловиться на границе.

**Рекомендация.** Валидировать ответы на границе: в `httpRepositories` оборачивать вызовы
`schema.parse(...)` (или `safeParse` с `ApiError`). И сделать единый источник типов —
выводить доменные типы из Zod (`type Post = z.infer<typeof postSchema>`), убрав дублирование
`Post` ↔ `PostDto`.

## 4. Качество кода и технический долг (среднее)

### 4.1. Нет глобальной обработки `401` (протухание токена) 🟠

`httpClient.ts` бросает `ApiError(status)`, `QueryProvider` на ошибку мутации показывает тост,
но **нет** перехвата `401 → автоматический logout / открытие auth-overlay`. Токен живёт в
`localStorage`; при его протухании пользователь будет получать молчаливые ошибки запросов.

**Рекомендация.** Централизованный обработчик `401` (в `httpClient` или через
`QueryCache.onError`): чистить сессию и открывать auth-overlay. Инфраструктура для этого уже
есть (`AuthProvider.logout`, `openAuthOverlay`).

### 4.2. Дублирование заглушки AI в `seed` и `msw`

`getGlobalReply` + сборка истории сообщений почти идентично продублированы в
`seedRepositories.ts` и `msw/handlers.ts`. После рефактора 3.1 это уйдёт в одно место.

### 4.3. Безопасность хранения токена (осознанный trade-off)

Bearer-токен в `localStorage` уязвим к XSS. Для статического SPA (`output: "export"`) это
типичный компромисс, но если бэкенд позволит — `httpOnly`-cookie безопаснее. Отметить как
сознательное решение, а не упустить.

### 4.4. Мутации вложенных коллекций шлют весь массив

`usePostNoteMutations.ts` отправляет `patch: { notes }` целиком при правке одной заметки.
Для local-first нормально, но для бэкенда это лишний трафик и гонки. Возможный переход на
гранулярные эндпоинты (`POST /posts/:id/notes`) — на будущее, не срочно.

## 5. Мелочи / nice-to-have

- **Покрытие тестами ~6.6% файлов (35/528).** Логика (`lib`, stores, schemas) покрыта хорошо;
  стоит точечно добавить тесты на репозитории `http`-режима (после внедрения Zod-валидации) и
  на реконсиляцию ID в мутациях.
- **Хардкод demo-меток** (`"OpenAI / gpt-4o"`, `"Perplexity / search-api"`) в заглушках —
  уедет вместе с рефактором 3.1.
- **`date: "сейчас"`** как литерал в нескольких местах — устранится при переходе на ISO (3.2).

## 6. План действий (по приоритету)

| # | Задача | Приоритет | Влияние на цель «не трогать фронтенд» |
|---|--------|:---------:|----------------------------------------|
| 1 | Вынести AI-ответы за `AssistantRepository` (убрать `setTimeout`/`replies.ts` из UI) | 🔴 Высокий | Снимает главный блокер |
| 2 | Перевести все `date` на ISO-8601, форматирование — в UI-функции | 🔴 Высокий | Снимает блокер контракта |
| 3 | Унифицировать ID (строковый/UUID) + реконсиляция серверного ID в `create` | 🟠 Средний | Корректные оптимистичные апдейты |
| 4 | Включить Zod-валидацию ответов на границе + единый источник типов из схем | 🟠 Средний | Безопасность контракта в рантайме |
| 5 | Глобальный обработчик `401` → logout/overlay | 🟠 Средний | Корректное протухание сессии |
| 6 | Дедуп заглушки AI (`seed`/`msw`), тесты на http-репозитории | 🟢 Низкий | Гигиена |

**Резюме.** Архитектурный «скелет» — отличный и его трогать не нужно. Чтобы достичь заявленной
цели «фронтенд готов к бэкенду без правок», нужно закрыть 4 протечки контракта данных
(разделы 3.1–3.4): это локальные, хорошо очерченные изменения, а не переписывание. После них
платформа действительно собирается в единый пазл, где бэкенд подключается сменой реализаций
репозиториев и двумя env-переменными.
