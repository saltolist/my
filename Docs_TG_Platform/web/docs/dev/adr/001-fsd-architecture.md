# ADR-001: Feature-Sliced Design как архитектурная методология

## Статус
✅ Принято

## Контекст

Проект — среднего размера SPA с несколькими взаимосвязанными доменами (посты, чаты, заметки, аналитика, профиль) и необходимостью параллельной разработки. Нужна методология, которая:
- Явно запрещает спагетти-зависимости между модулями
- Масштабируется при росте команды
- Обеспечивает изоляцию фичей

## Рассмотренные варианты

- **Feature-Sliced Design** — строгая слоистая методология с правилами импортов
- **"Модульная" структура** — папки по доменам без правил зависимостей
- **"Плоская" структура** — все компоненты в одной директории

## Решение

Принят **Feature-Sliced Design**. Слои (сверху вниз):

```
app → screens → widgets → features → entities → shared
```

Каждый слой может импортировать только из нижележащих.  
Правило зависимостей контролируется `eslint-plugin-boundaries`.

Импорты используют алиас `@/` = `src/`, исключая относительные пути между слоями.

## Последствия

- Каждый слайс изолирован и тестируемо независимо
- Добавление новой фичи не затрагивает другие слайсы
- `eslint-plugin-boundaries` ловит нарушения на этапе lint/CI
- Небольшое увеличение шаблонного кода — компромисс за предсказуемость
