# ADR-004: ISO-8601 даты в модельном слое

## Статус
✅ Принято

## Контекст

Исторически даты в модели хранились как человекочитаемые строки: `"28 апр · 14:22"`, `"сейчас"`, `"только что"`, `"1 мая"`. Это создавало критические проблемы:

1. **Нет машинной обработки** — невозможно сортировать, сравнивать, вычислять дельты.
2. **Локализация в модели** — бизнес-логика смешана с представлением.
3. **Несовместимость с бэкендом** — любой API возвращает ISO-8601, а не локализованные строки.
4. **Хрупкий парсинг** — функция `parsePostDateTime` содержала regex для русских месяцев.

## Рассмотренные варианты

- **Оставить читаемые строки** — несовместимо с любым реальным API
- **Unix timestamp (число)** — машинный, но не читается в логах/базе
- **ISO-8601 строки** — стандарт де-факто, читаемо и машинно

## Решение

Все даты в модели хранятся как ISO-8601: `"2026-04-28T14:22:00.000Z"`.

`parsePostDateTime` упрощён до одной строки:
```typescript
export function parsePostDateTime(raw: string | undefined | null): Date | null {
  if (!raw) return null;
  const d = new Date(raw);
  return Number.isFinite(d.getTime()) ? d : null;
}
```

Форматирование для UI вынесено в отдельный хелпер:
```typescript
export function formatStoredDate(raw: string | undefined | null): string {
  const d = parsePostDateTime(raw?.trim());
  if (!d) return raw ?? "";
  if (isDateOnlyIso(raw)) return formatPostDateUtc(d);   // "28 апр"
  return formatPostDateTimeUtc(d);                        // "28 апр · 14:22"
}
```

Все компоненты, отображающие даты, вызывают `formatStoredDate(date)`.

## Последствия

- Модель полностью совместима с любым бэкенд API
- Можно сортировать посты по дате строковым сравнением ISO
- Форматирование даты — ответственность UI-слоя, а не модели
- UTC-интерпретация дат в seed-данных обеспечивает консистентность
