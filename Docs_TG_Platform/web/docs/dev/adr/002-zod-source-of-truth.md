# ADR-002: Zod как единый источник доменных типов

## Статус
✅ Принято

## Контекст

В проекте существовала проблема дублирования: доменные типы (`Post`, `LocalNote`, `GlobalChat` и др.) объявлялись вручную в `shared/types/index.ts`, а отдельно — Zod-схемы для валидации. При изменении модели данных нужно было обновлять оба места, что приводило к рассинхронизации.

Дополнительно, HTTP-репозиторий возвращал результаты без валидации (`apiRequest<Post[]>`), принимая данные от сервера на веру.

## Рассмотренные варианты

- **Ручные типы + Zod отдельно** — текущее состояние, требует двойного обновления
- **Zod как источник типов** — типы выводятся через `z.infer<typeof schema>`
- **Codegen из OpenAPI** — внешний источник, добавляет зависимость от бэкенда

## Решение

Все доменные типы теперь выводятся из Zod-схем:

```typescript
// shared/api/schemas/post.ts
export const postSchema = z.object({ id: z.string(), ... });
export type Post = z.infer<typeof postSchema>;
```

`shared/types/index.ts` только реэкспортирует типы из схем:
```typescript
export type { Post, LocalNote, ... } from "@/shared/api/schemas";
```

HTTP-репозиторий парсит каждый ответ:
```typescript
list: () => apiRequest<unknown>(apiV1Path("posts"))
  .then((data) => postsListSchema.parse(data))
```

Невалидный ответ бэкенда вызовет `ZodError` — ошибка видна немедленно.

## Последствия

- Единое место правды для типов — только Zod-схемы
- Автоматическая проверка совместимости данных от сервера
- Типы и валидация всегда синхронизированы
- `ZodError` с детальным описанием вместо неочевидных runtime-ошибок
