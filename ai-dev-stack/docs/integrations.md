# Интеграции

Раздел описывает, как я подключаю внешние системы к AI-платформе без потери управляемости и качества.

Цель: построить интеграционный слой, который масштабируется по числу каналов, но сохраняет единые правила безопасности, наблюдаемости и продуктовой логики.

## Принцип интеграции

Базовый подход: `API-first + event-driven`.

Это означает:

- у каждого канала есть явный контракт входа/выхода;
- внутренняя логика платформы не зависит от конкретного клиента (web, Telegram, CRM и т.д.);
- события и webhooks обрабатываются через общий orchestration-слой.

## Принципы интеграционного дизайна

- **Единый внутренний контракт**: все внешние payload приводятся к общему доменному формату.
- **Каналы как адаптеры**: Telegram, CRM, email и web не дублируют бизнес-логику.
- **Asynchronous by default**: долгие и нестабильные операции выносятся в очередь/воркеры.
- **Idempotency everywhere**: повторная доставка событий не должна ломать состояние.
- **Observable integration flow**: каждый шаг интеграции трассируется и дебажится.

## Паттерн подключения каналов

Обычно любой внешний канал подключается по одинаковой схеме:

1. **Adapter layer**: нормализация payload канала в единый внутренний формат.
2. **Policy layer**: проверка прав, лимитов, tenant-контекста и маршрутизации.
3. **AI execution layer**: сбор контекста, вызов модели/агента, tool use.
4. **Delivery layer**: возврат ответа в формат конкретного канала.

Такой подход позволяет масштабировать число каналов без дублирования бизнес-логики.

### Минимальный внутренний контракт события

Чтобы интеграции были управляемыми, входящий запрос обычно приводится к структуре:

- `event_id` / `request_id`;
- `channel` и `event_type`;
- `tenant_id`, `user_id`, `space_id` (если применимо);
- `payload` (нормализованные данные);
- `timestamp` и `source_metadata`;
- `idempotency_key` для защиты от повторов.

## Синхронные и асинхронные интеграции

Практическое разделение:

- **синхронный контур**: короткие запросы с жестким SLA по latency;
- **асинхронный контур**: долгие задачи, batch-обогащения, внешние ретраи и нестабильные API.

Правило маршрутизации:

- если операция может превысить UX-SLA, она уходит в async pipeline;
- синхронный endpoint сразу возвращает `accepted` + correlation id;
- финальный результат доставляется callback/webhook/notification-сценарием.

## Webhooks и асинхронные сценарии

Webhooks используются для:

- запуска фоновых агентных задач;
- синхронизации состояний между внешними системами и платформой;
- реакций на бизнес-события (создание лида, смена статуса, новый документ и т.д.).

Практики надежности:

- идемпотентность обработчиков;
- retry с ограничением количества попыток;
- дедупликация входящих событий;
- журналирование входа/выхода и причин ошибок.

Дополнительно для production:

- подпись webhook и защита от replay-атак;
- rate-limit на ingress;
- dead-letter queue для необработанных событий;
- ручной replay событий из failed-очереди;
- трассировка `webhook -> task -> результат` по единому correlation id.

## Очереди и фоновые обработчики

Для Python-проектов типовой async-контур: `Celery + Redis`.

Что обычно уходит в очередь:

- обработка входящих webhook-событий;
- вызовы внешних API с ретраями;
- длительные AI-задачи, enrichment, классификация;
- sync-джобы и reconciliation между системами.

Базовые требования к задачам:

- идемпотентность и side-effect safety;
- bounded retries + exponential backoff;
- таймауты и отмена зависших задач;
- отдельные очереди по приоритетам (`critical/default/batch`);
- мониторинг queue depth, failure rate, retry rate, processing time.

## Версионирование контрактов

Интеграции неизбежно эволюционируют, поэтому контракт должен быть версионируемым:

- versioning payload-схем (`v1`, `v2`, ...);
- backward compatibility на переходный период;
- явная deprecation policy для старых версий;
- contract tests между адаптером и core-логикой.

Практика:

- breaking changes не выкатываются "молча";
- для критичных каналов используются canary-обновления;
- migration window фиксируется заранее.

## Мультиканальность

Один и тот же ассистент может работать сразу в нескольких интерфейсах:

- веб-приложение;
- мессенджер (например, Telegram);
- внутренние кабинеты команды.

Ключевой принцип: логика принятия решения едина, отличаются только transport- и presentation-слои.

Практические правила мультиканальности:

- единый доменный слой и policy engine для всех каналов;
- channel-specific только формат ввода/вывода и UX-ограничения;
- одинаковые guardrails и права доступа независимо от канала;
- единые telemetry и eval-метрики для сравнения качества между каналами.

## Интеграции с внешними бизнес-системами

Наиболее частые типы внешних систем:

- CRM/ERP (лиды, сделки, статусы, комментарии);
- сервисы коммуникаций (Telegram, email, чат-платформы);
- документы/knowledge systems (Notion, Google Docs, Confluence);
- биллинг и платежные системы;
- аналитика и продуктовые события.

Базовая стратегия:

- integration adapter изолирует внешние API-особенности;
- core-логика опирается на стабильный внутренний контракт;
- ошибки и нестабильность внешних API не "протекают" в доменную модель.

## Согласованность данных и reconciliation

Интеграции между несколькими системами требуют явной политики согласованности:

- где source-of-truth для каждой сущности;
- кто может менять статус и в каком порядке;
- как обрабатываются race conditions и конфликтующие обновления;
- как выполняется nightly/periodic reconciliation.

Минимальные механизмы:

- optimistic locking или version check;
- event ordering (где критично);
- idempotent upsert операции;
- отчеты по рассинхронизации и автоматический repair-пайплайн.

## Безопасность интеграций

Минимальный набор требований:

- подпись и верификация webhooks;
- scoped API-ключи и ротация секретов;
- аудит критичных вызовов инструментов;
- ограничение доступа к данным на уровне tenant/user.

Дополнительные требования для enterprise-контуров:

- least-privilege сервисные аккаунты;
- mTLS или private networking для внутренних интеграций;
- шифрование данных in transit и at rest;
- redaction чувствительных полей в логах интеграционного слоя;
- контроль трансграничной передачи данных по политике проекта.

## Наблюдаемость интеграционного слоя

Ключевые метрики:

- success/error rate по каналам и типам событий;
- p50/p95 latency на каждом шаге pipeline;
- queue lag и время обработки фоновых задач;
- доля retries, dead-letter и manual replays;
- стоимость интеграционного контура + связанная LLM-стоимость.

Operational минимумы:

- централизованный structured logging;
- correlation id в каждом запросе/событии;
- алерты на деградацию каналов и внешних API;
- dashboard по health каждого интеграционного адаптера.

## Тестирование интеграций

Минимальный контур тестирования:

- contract tests на вход/выход адаптеров;
- e2e тесты ключевых мультиканальных сценариев;
- sandbox/stage проверки для внешних API;
- chaos/retry tests на нестабильность внешнего канала.

Перед релизом интеграции проверяю:

- корректность прав доступа и tenant-изоляции;
- идемпотентность обработчиков и защиту от дублей;
- rollback план для изменения контракта;
- наблюдаемость и алерты по критичным каналам.

## Стоимость и масштабирование

Интеграционный слой масштабируется по каналам и событиям, поэтому важен контроль стоимости:

- лимиты на входящий event-rate по каналу;
- батчирование там, где это не ломает SLA;
- кэширование стабильных внешних данных;
- отключаемые фичи и degradation-mode при всплесках нагрузки.

Приоритет оптимизации:

1. уменьшить лишние внешние вызовы;
2. стабилизировать retries/dedup и async orchestration;
3. затем масштабировать compute/воркеры.

## Что считаю анти-паттернами

- копирование бизнес-логики в каждый канал;
- отсутствие idempotency и dedup на webhook-входе;
- прямой вызов LLM из channel-adapter без policy/orchestration слоя;
- отсутствие versioning контрактов при активной интеграционной эволюции;
- "черный ящик" без трассировки и correlation id.

## Практический checklist запуска

Перед запуском нового интеграционного канала проверяю:

- определен внутренний контракт и schema-version;
- настроены webhook verification, dedup и retry-политики;
- async-контур (очередь/воркеры) покрывает долгие операции;
- включены алерты и dashboard по каналу;
- есть rollback и migration plan для будущих изменений контракта.
