# Тестирование

Раздел про контроль качества разработки в целом: что и как проверяется до и после релиза.

Цель: сделать качество системы повторяемым и управляемым, чтобы решения о релизе принимались по данным, а не по впечатлению от демо.

## Принципы тестирования в целом

- **Test pyramid + extensions**: база качества строится на классических тестах, а специализированные проверки добавляются сверху.
- **Scenario-first**: тесты строятся вокруг реальных пользовательских сценариев и бизнес-рисков.
- **Regression by default**: каждое значимое изменение проходит регрессию.
- **Quality/latency/cost вместе**: релиз оценивается сразу по качеству, скорости и стоимости.
- **Production feedback loop**: реальные сбои и неудачные кейсы пополняют тестовые наборы.

## Что тестирую

Система проверяется как набор связанных слоев:

- **Core backend**: доменная логика, права, статусы, транзакции, API-контракты.
- **Frontend/UX**: корректность экранов, состояний, форм, навигации и ошибок.
- **Интеграции**: webhooks, адаптеры, синхронизация с внешними системами.
- **Данные**: миграции, целостность, source-of-truth, восстановление.
- **Инфраструктура**: очереди, воркеры, конфигурация окружений, надежность.
- **Безопасность**: доступы, валидация, audit trail, контроль рискованных действий.
- **AI-слой**: evals, retrieval/memory, tool use, guardrails (если AI есть в продукте).

## Базовый контур качества

Этот контур обязателен независимо от наличия AI:

- функциональные тесты (`unit`, `integration`, `e2e`) по критичным сценариям;
- контрактные тесты API и событийных интерфейсов;
- нефункциональные проверки (latency, надежность, устойчивость под нагрузкой);
- проверки безопасности и прав доступа;
- release gates по качеству, latency и стоимости эксплуатации.

## Уровни тестирования

- **Unit**: бизнес-логика, валидаторы, маршрутизация, policy-функции.
- **Integration**: взаимодействие сервисов, БД, очередей и внешних API.
- **E2E**: пользовательские потоки от входа до финального результата.
- **Contract**: совместимость API/event payload между модулями.
- **Non-functional**: производительность, надежность, ресурсы.
- **Security**: права, ограничения, валидация входов и рискованные операции.

## Специализированный AI-контур

AI-часть не заменяет общий QA, а дополняет его:

- `AI evals` по ключевым сценариям;
- регрессия качества после изменений модели/промптов/роутинга;
- проверки retrieval/memory;
- валидация tool use и safety-ограничений.

## Базовый baseline перед AI-gates

До AI-evals обязательно проходят базовые проверки системы:

- backend-контракты не сломаны;
- UI корректно обрабатывает статусы и ошибки;
- интеграционные адаптеры проходят контрактные тесты;
- миграции и rollback-стратегия валидны;
- async-контур стабилен под нагрузкой.

AI-оценка включается после подтверждения базовой функциональности.

## Evals как quality gate

Минимальный eval-набор обычно покрывает:

- основные product use-cases;
- сложные edge-cases и неоднозначные запросы;
- негативные сценарии (недостаточно данных, конфликт источников, запрет действия);
- критичные сценарии tool use.

Для каждого кейса фиксируются:

- входные данные и ожидаемый outcome;
- допустимые границы качества;
- критерии провала;
- теги сценария (domain, risk, space_type, channel).

Практика обновления:

- новые пользовательские провалы добавляются в eval-набор в ближайшем цикле;
- дубли и устаревшие кейсы чистятся по расписанию;
- eval-набор версионируется как часть репозитория.

## Регрессия изменений

Любое значимое изменение проверяю по одному шаблону:

1. baseline-замер на текущей версии;
2. запуск кандидатной версии на том же наборе;
3. сравнение качества, latency и стоимости;
4. решение: release / fix / rollback.

Отдельно считаю значимыми:

- смену модели или параметров модели;
- изменение prompt/router-конфигурации;
- изменения retrieval/memory-пайплайна;
- обновление tool contracts и оркестрации;
- изменения в критичных API/интеграционных контрактах.

## Контекст, память и tool use

Для AI-сценариев отдельно валидирую:

- полноту и релевантность собранного контекста;
- приоритет source-of-truth при конфликте источников;
- корректность `space-aware` фильтрации и tenant-изоляции;
- поведение memory write policy;
- tool use: права до действия, идемпотентность, timeout/retry/fallback.

## Нефункциональные проверки

Помимо функциональной корректности:

- p95 latency по ключевым сценариям;
- устойчивость под нагрузкой и burst-событиями;
- надежность async-контуров (очереди/воркеры);
- стоимость (`cost per successful scenario`) и budget guardrails.

## Каналы и интеграции

Если сценарий мультиканальный, проверка идет по каждому каналу:

- корректность адаптера и нормализации payload;
- сохранение единых policy-правил между web/Telegram/CRM;
- webhook idempotency и dedup;
- трассировка `request -> task -> result` через correlation id.

## Human-in-the-loop

Для критичных бизнес-операций внедряются ручные точки подтверждения:

- review чувствительных действий;
- валидация итогового ответа перед внешней отправкой;
- разбор спорных кейсов для улучшения eval-наборов.

Критерии обязательного HITL:

- высокая стоимость ошибки;
- юридические/финансовые последствия;
- действия с внешними системами и необратимыми side-effects.

## Release gates и решение о выкате

Релиз изменения допускается, если одновременно выполнены условия:

- функциональная целостность системы подтверждена;
- качество не ниже установленного порога;
- latency в пределах SLA;
- стоимость в пределах бюджета;
- safety-проверки и права доступа не нарушены.

Если любой gate провален:

- изменение откладывается или ограничивается флагом;
- запускается короткий цикл исправлений;
- повторный релиз только после повторной регрессии.

## Production-контроль качества

После релиза качество отслеживается непрерывно:

- мониторинг ошибок, latency и стоимости;
- отбор неудачных кейсов в наборы регрессии;
- регулярное обновление тест-кейсов по фактическому поведению пользователей.

Для не-AI функционала дополнительно:

- анализ дефектов UI/API и скорости их исправления;
- контроль стабильности интеграционных каналов;
- проверка целостности данных после релизов и миграций.

## Набор ключевых метрик

Продуктовые:

- task success rate;
- доля ручных правок;
- CSAT/feedback score;
- conversion draft -> applied.

Платформенные:

- crash-free rate и availability по ключевым сервисам;
- defect escape rate (дефекты, дошедшие до продакшена);
- webhook/integration success rate;
- миграции без инцидентов и успешность restore-check.

Операционные:

- p50/p95 latency;
- error/fallback rate;
- queue lag (для async-сценариев);
- cost per successful scenario.

AI-метрики:

- доля ответов с корректными источниками;
- recall/релевантность retrieval;
- false-memory-rate (для memory-сценариев);
- success rate tool use без ручного вмешательства.

## Что считаю анти-паттернами

- ручное тестирование "на глаз" без автоматического контура;
- релиз без регрессии после значимых изменений;
- игнорирование функциональных регрессий ради локального улучшения одного компонента;
- оценка только качества без latency и стоимости;
- отсутствие safety-проверок для рискованных действий;
- игнорирование production-фидбека в обновлении тестовых наборов.

## Практический checklist запуска

Перед релизом изменения проверяю:

- пройдены базовые проверки backend/frontend/integrations/data;
- пройдены unit/integration/e2e для затронутых модулей;
- проверены права доступа и risk-controls;
- latency и стоимость в пределах целевых порогов;
- есть rollback и feature-flag план;
- если AI есть в сценарии - обновлен и прогнан релевантный eval-набор;
- после релиза включен мониторинг и разбор неудачных кейсов.
