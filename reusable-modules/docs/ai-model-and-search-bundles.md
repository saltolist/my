# AI Model & Search Bundles

## Идея кейса

`AI Model & Search Bundles` — это инструмент конфигурации AI-ответов, где пользователь управляет связками "модель + AI-поисковик" и получает несколько версий ответа в одном интерфейсе.

Кейс демонстрирует не "еще один чат", а управляемый контур генерации с явным сравнением результатов разных связок.

## Какую проблему решает

В большинстве AI-интеграций ответ формируется одной моделью в одном режиме. Из-за этого:
- нет возможности быстро сравнить качество разных моделей;
- сложно понять, дает ли AI-поисковик прирост для конкретного запроса;
- пользователь ограничен одной версией ответа и не контролирует стратегию генерации.

`AI Model & Search Bundles` решает это через явные конфигурации связок и параллельную генерацию альтернатив.

## Ключевые возможности

- Выбор одной или нескольких моделей для генерации ответа.
- Создание и сохранение связок "модель + AI-поисковик".
- Опциональный классификатор необходимости поиска, который определяет, нужно ли подключать AI-веб-поиск для конкретного запроса.
- Параллельная генерация нескольких ответов, если выбрано несколько моделей или связок.
- Переключение альтернативных ответов в интерфейсе без нового запроса.
- Формулировщик поискового запроса для AI-веб-поиска (query rewriter), который учитывает контекст и историю чата и улучшает качество retrieval.
- Кэш найденных результатов по embedding-представлению сформулированного поискового запроса.

## Архитектурный контур

1. Пользователь выбирает модели или готовые связки "модель + AI-поисковик".
2. Система получает исходный запрос пользователя.
3. При включенном классификаторе система определяет, нужен ли веб-поиск для текущего запроса.
4. Для связок с поиском (или когда классификатор определил необходимость поиска) query rewriter формирует оптимальный веб-поисковый запрос с учетом контекста чата.
5. По embedding-представлению проверяется кэш релевантных результатов поиска.
6. Для активных конфигураций запускается генерация ответов.
7. Пользователь видит основной и альтернативные ответы и может переключаться между ними.

## Что показывается в демо

- Запрос с одной моделью без поиска (базовый сценарий).
- Запрос, где классификатор отключает веб-поиск как избыточный.
- Тот же запрос в двух разных связках "модель + AI-поисковик" с визуальным сравнением результата.
- Переключение между альтернативными ответами в один клик.
- Повторный запрос с попаданием в embedding-кэш для ускоренного ответа.

## Технологический фокус

- Multi-model configuration
- Search bundle management
- Search necessity classification
- Query rewriting for web search
- Embedding cache for retrieval
- Multi-model strategy
- Alternative response switching

## Практическая ценность

Инструмент можно встроить в веб-ассистента или Telegram-бота как слой управляемой генерации: бизнес получает гибкость в выборе AI-стека, а пользователь — понятный механизм сравнения и выбора лучшего ответа.

## Интеграция с AI Orchestrator Layer

Компонент может работать автономно со своей внутренней обработкой запроса: классификатор необходимости поиска, web query rewriter и запуск связок генерации.

Если в системе используется внешний обработчик запросов (например, `AI Orchestrator Layer`), входная логика компонента может быть делегирована ему, а `AI Model & Search Bundles` работает как специализированный исполнительный слой генерации.

## Интеграция с Data Depersonalization Layer

Для сценариев с пользовательскими данными рекомендуется подключать `Data Depersonalization Layer` как обязательный этап перед LLM-вызовом.

В production-контуре предпочтительна связка `Data Depersonalization Layer` + `AI Orchestrator Layer`: она дает более удобную централизованную настройку правил очистки и маршрутизации по сравнению с использованием только встроенных (родных) классификаторов и локальной оркестрации компонента.
