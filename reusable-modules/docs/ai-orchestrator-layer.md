# AI Orchestrator Layer

## Идея кейса

`AI Orchestrator Layer` — это опциональный метаслой, который берет на себя обработку входящего запроса: анализ, выбор стратегии и координацию модулей платформы в единый сценарий исполнения.

Ключевой принцип архитектуры: каждый модуль системы остается самостоятельным и умеет работать с запросом напрямую, а оркестратор подключается поверх них, когда нужен централизованный контроль и сложные составные сценарии.

Практически это означает, что общие функции обработки запроса (классификация, формирование web/RAG-запросов, разбор агентных команд) есть внутри модулей, но при multi-module конфигурации могут быть централизованы в оркестраторе как в едином процессоре пользовательских сообщений.

## Какую проблему решает

Когда в платформе много независимых AI-модулей, без объединяющего слоя появляются:

- дублирование логики обработки запроса в разных местах;
- сложность настройки сквозных сценариев;
- неравномерное качество ответов при переключении между режимами;
- рост издержек на сопровождение при добавлении новых модулей.

`AI Orchestrator Layer` упрощает масштабирование: он стандартизирует вход в систему и маршрутизирует запрос в нужный модуль или цепочку модулей.

## Ключевые возможности

- Анализ входящего запроса и определение сценария обработки.
- Классификация намерения пользователя для выбора подходящего пути исполнения.
- Обработка агентных команд и передача их в `Agent Actions Layer`.
- Поддержка агентных команд с retrieval-этапом, когда для выполнения команды требуется RAG и/или AI-веб-поиск.
- Централизованный запуск составных сценариев между модулями.
- Два формулировщика запроса: для AI-веб-поиска и для RAG-поиска по внутренней базе, оба с учетом контекста чата.
- Поддержка гибридного режима, где web-query rewriter и rag-query rewriter работают совместно в одном сценарии.
- Опциональные правила по интентам: веб-поиск можно включать только для выбранных типов запросов.
- Два режима исполнения: последовательный (дешевле и более контролируемо) и параллельный (быстрее при сложных запросах).
- Пользовательский выбор стратегии исполнения с явным компромиссом "скорость vs стоимость".
- Поддержка fallback-стратегий, если выбранный модуль недоступен или вернул слабый результат.
- Режим "orchestrator on/off": модули работают как самостоятельно, так и под управлением оркестратора.

## Архитектурный контур

1. В систему поступает пользовательский запрос.
2. Orchestrator нормализует запрос и определяет выбранный пользователем режим исполнения.
3. В последовательном режиме сначала определяется интент и тип команды, затем запускаются формулировщики контекста (по умолчанию web + RAG, либо по интент-правилам пользователя) с учетом контекста чата.
4. В параллельном режиме анализ интента и формирование веб/RAG-запросов с учетом контекста чата запускаются одновременно для ускорения ответа.
5. Выбирается план исполнения: один модуль или цепочка модулей.
6. Для агентных команд при необходимости выполняется retrieval-контур (RAG и/или web search) перед запуском действия.
7. Orchestrator передает задачу в целевой модуль и собирает результат.
8. Пользователь получает ответ и технический trace выбранного маршрута.

## Что показывается в демо

- Один и тот же запрос в двух режимах: прямой вызов модуля и вызов через оркестратор.
- Сравнение последовательного и параллельного режимов на одном запросе (время ответа и стоимость исполнения).
- Сценарий классификации интента с выбором разных путей исполнения.
- Сценарий агентной команды, переданной из оркестратора в `Agent Actions Layer`.
- Сценарий агентной команды с дополнительным retrieval-контуром: перед действием выполняется RAG и/или веб-поиск.
- Гибридный сценарий, где web-query rewriter и rag-query rewriter работают вместе в одном запросе.
- Настраиваемый сценарий, где веб-поиск включается только для выбранных интентов, заданных пользователем.
- Fallback-ветка: автоматическое переключение на альтернативный модуль.

## Технологический фокус

- Intent routing
- Agent command handling
- Workflow coordination
- Hybrid Web/RAG query rewriting
- Optional meta-layer architecture
- Fallback and resiliency logic

## Практическая ценность

Оркестратор делает платформу гибкой для кастомизации под конкретного заказчика: можно оставить модульный режим для простых сценариев или включить централизованную координацию с настраиваемым режимом исполнения. Пользователь сам выбирает, что для него приоритетнее в конкретном потоке — минимальная задержка или оптимизация стоимости.
