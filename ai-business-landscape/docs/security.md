# Безопасность

Раздел о практической безопасности AI-продуктов: какие угрозы специфичны для LLM-систем, где чаще всего происходят утечки и как строить защиту так, чтобы она работала в продакшне, а не только в презентации.

Цель: снизить вероятность и стоимость инцидентов через архитектуру, процессы и эксплуатационные guardrails.

## Базовый принцип

LLM не должен считаться доверенным компонентом. Он может:

- неверно интерпретировать контекст;
- быть уязвимым к инъекциям;
- генерировать рискованные действия при неточной постановке.

Поэтому безопасность строится по модели `assume breach`: ограничиваем полномочия, проверяем критичные действия, фиксируем аудит.

## Ключевые классы угроз

### 1) Prompt injection

Типовые сценарии:

- пользователь или внешний источник вставляет инструкции, меняющие поведение агента;
- retrieval подтягивает вредоносный контент, который выглядит как "системные правила";
- tool-calling запускается на основе подменённых указаний.

Меры:

- жёсткое разделение trusted instructions и untrusted data;
- фильтрация и нормализация retrieved контента;
- policy-проверка до любого side-effect действия.

### 2) Data leakage

Риски:

- смешение tenant-данных;
- логирование чувствительной информации без маскирования;
- отправка приватных данных внешним провайдерам без контролей.

Меры:

- data classification и redaction до вызова модели;
- tenant isolation на уровне запроса, контекста и storage;
- минимизация retention и контролируемая политика экспорта.

### 3) Excessive agency / over-permission

Риск возникает, когда агенту дают слишком широкие полномочия:

- доступ к действиям вне текущего сценария;
- отсутствие подтверждения для необратимых операций;
- выполнение цепочки инструментов без ограничений.

Меры:

- principle of least privilege;
- capability allowlist для каждого use-case;
- обязательный confirmation gate для рискованных шагов.

### 4) Model reliability risk

Даже без атак система может быть небезопасной:

- hallucinations в критичных доменах;
- нестабильность качества после обновлений;
- непредсказуемое поведение при edge-cases.

Меры:

- eval-regression перед релизом;
- deterministic проверки структурированных выходов;
- fallback на безопасный режим при низкой уверенности.

## Архитектурные guardrails

Практический минимум для production:

- **Policy Engine** между LLM и внешними действиями;
- **Tool firewall**: каждый вызов инструмента валидируется;
- **Context boundary enforcement**: запрет кросс-tenant смешивания;
- **Output validation**: схема, бизнес-правила, лимиты;
- **Audit trail** для всех критичных операций.

Важно: LLM формирует предложение действия, но не выполняет его напрямую.

## Контроль данных и приватности

Надёжный контур данных включает:

- классификацию (`public/internal/confidential/restricted`);
- redaction PII/секретов до отправки в модель;
- шифрование in-transit и at-rest;
- ограничение сроков хранения логов и контекстов;
- раздельные ключи и доступы по окружениям.

Если невозможно доказать, где хранятся и как удаляются данные, это не enterprise-ready решение.

## Безопасный tool use

Перед любым внешним действием проверяю:

- право роли на конкретное действие;
- валидность параметров и диапазонов;
- идемпотентность операции;
- timeout/retry и graceful failure;
- необходимость human approval.

Критичное правило: "прочитать можно шире, чем изменить". Write-права всегда уже, чем read-права.

## Human-in-the-loop для рискованных действий

HITL обязателен, если:

- действие финансово или юридически значимо;
- операция необратима;
- есть риск влияния на внешнего клиента или партнёра.

Формат:

- модель предлагает действие и обоснование;
- человек подтверждает или отклоняет;
- решение и контекст фиксируются в аудит-логе.

## Операционная безопасность

После запуска нужно контролировать:

- anomalous prompt patterns;
- всплеск fallback/ошибок tool use;
- признаки кросс-tenant аномалий;
- отклонение поведения после обновления модели.

Без мониторинга даже хорошие guardrails деградируют незаметно.

## Incident response для AI-систем

Минимальный playbook:

1. обнаружение и классификация инцидента;
2. containment (ограничение прав, отключение каналов/tooling);
3. анализ причины (prompt, данные, интеграция, модель);
4. исправление и проверка регрессией;
5. обновление политик, evals и runbook.

Отдельно важно иметь `safe mode`: режим, где система продолжает базовую работу с минимальным риском.

## Security by design в процессе разработки

Практики, которые должны идти "по умолчанию":

- threat modeling на этапе проектирования сценария;
- security review для новых инструментов и интеграций;
- pre-release gates по доступам и политике данных;
- регулярные tabletop-упражнения по инцидентам.

## Метрики безопасности

Ключевые операционные метрики:

- `blocked risky actions rate`;
- `policy violation rate`;
- `time to detect` и `time to contain`;
- доля сценариев с корректным tenant-boundary enforcement;
- `security regression rate` после изменений.

## Что считаю анти-паттернами

- доверять "честности" промпта без независимой policy-проверки;
- смешивать данные клиентов в общем контексте;
- давать агенту write-доступ без подтверждений;
- логировать секреты и PII "для удобства дебага";
- выпускать изменения модели без security-regression.

## Практический checklist перед релизом

Перед запуском нового AI-сценария проверяю:

- определены trust boundaries и классы данных;
- настроены role-based и action-based ограничения;
- внедрены validation и confirmation для рискованных действий;
- есть audit trail по критичным событиям;
- прогнаны security-тесты и негативные сценарии;
- подготовлен incident playbook и safe mode fallback.
