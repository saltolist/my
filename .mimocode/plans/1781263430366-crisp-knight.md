# Code Review: tg-platform-web

## Executive Summary

Проект **tg-platform-web** — это AI-powered SPA для управления Telegram-каналами (лента, чаты, заметки, аналитика, посты). Технологический стек: Next.js 16 (App Router, static export), React 19, TypeScript strict, Zustand, TanStack Query, Tailwind CSS 4, shadcn/ui, Zod. Архитектура — **Feature-Sliced Design (FSD)** с ESLint enforcement. Проект находится на вехе M2 (Shell complete), примерно 60-70% кодовой базы построено.

**Общая оценка: выше среднего для проекта на этом этапе.** Сильная архитектурная база, хорошая типизация, интеллектуальные паттерны (Repository, 3-mode data source). Ключевые зоны роста: отсутствие error boundaries, использование `window.alert/confirm`, fire-and-forget мутации, дублирование типов Zod/TypeScript.

## 1. Архитектура

### 1.1 Feature-Sliced Design — ✅ Сильно

FSD — правильный выбор для проекта этого масштаба. Импорт-границы严格落实лены через `eslint-plugin-boundaries` в `eslint.config.mjs:30-63`.

**Сильные стороны:**
- Чистая послойная структура: `app → screens → widgets → features → entities → shared`
- Каждый слой имеет `model/`, `ui/`, `lib/`, `index.ts` — единообразно
- Barrel exports (`index.ts`) создают чистые public API для каждого среза
- ESLint enforcement предотвращает архитектурный дрейф

**Рекомендации:**
- `composer-store.tsx` (React Context) живёт в `app/model/store/` — это единственный файл с `"use client"` и JSX в `model/`. Лучше вынести в `widgets/composer/model/` или `app/providers/`, так как это не纯粹 store
- Features слой пока минимален (5 срезов: delete-note, rename-note, manage-drafts, schedule-post, post-context-menu). По мере роста — следить за тем, чтобы business actions не мигрировали в widgets

### 1.2 Repository Pattern + 3-Mode Data Source — ✅ Отлично

**`shared/api/repositories.ts`** — чистые интерфейсы с типизированными методами. **`shared/api/createRepositories.ts`** — фабрика с 3 режимами: seed/msw/http.

**Сильные стороны:**
- Единый интерфейс для всех data source modes
- MSW-first разработка без бэкенда
- Zod-валидация DTO на границе API (`shared/api/schemas/`)
- `RepositoryProvider` inject через React Context

**Рекомендации:**
- `shared/api/schemas/` дублируют типы из `shared/types/index.ts`. Например, `PostDto` (из Zod inference) и `Post` (из types/index.ts) — идентичны. Решение: генерировать domain types из Zod schemas через `z.infer<>` и убрать ручные типы, либо наоборот — использовать Zod только для валидации, а types — как source of truth
- `httpClient.ts:50-54` — `undefined as T` — unsafe cast. Лучше `return undefined as unknown as T` или типизировать возвращаемый тип как `T | undefined`
- `createRepositories.ts:12-16` — модульный singleton (`cached`) без TTLS. В long-running dev session это нормально, но стоит добавить комментарий о предполагаемом lifecycle

### 1.3 State Management — ✅ Хорошо

Гибридный подход: Zustand для UI/ephemeral, TanStack Query для server state.

**Сильные стороны:**
- Чёткое разделение: Zustand = клиентское, Query = серверное
- Fine-grained selectors везде (`useNavigationStore((s) => s.feedSearch)`)
- `ui-store.ts` с `persist` + `partialize` — правильно сохраняет только `sidebarCollapsed`

**Рекомендации:**
- `composer-store.tsx` использует React Context вместо Zustand. Это осознанный выбор (нужен ref bridge +副作用), но увеличивает provider nesting. Рассмотреть Zustand store с `useRef` pattern для nav bridge
- `useUi()` в `ui-store.ts:101-137` — convenience hook который 13 раз вызывает `useUiStore()`. Каждый вызов создаёт подписку. Рекомендуется объединить в один `useUiStore(s => ({...}))` с shallow equality, либо оставить как есть если перфёрманс не проблема

### 1.4 Routing — ⚠️ Нужно внимание

`RouteSync` widget синхронизирует URL ↔ Zustand navigation. Это workaround для Next.js static export.

**Проблемы:**
- `navigation-store.ts` содержит screen-specific фильтры (`chatsTab`, `noteScope`, `feedSearch`, `analyticsPeriod`). Это нарушает单一 ответственность — navigation store стал "everything store"
- `post-navigation-store.ts` — отдельный store для post mode stack. Хорошо для изоляции, но сложность растёт: 2 navigation stores + URL sync + RouteSync widget

**Рекомендации:**
- Вынести фильтры (chatsTab, noteScope, feedSearch) в отдельные stores по feature, либо в URL search params
- Документировать контракт между RouteSync и stores — сейчас это implicit

## 2. Чистота кода

### 2.1 Типизация — ✅ Отлично

TypeScript strict mode, нет `any`, Zod-валидация.

**Сильные стороны:**
- `tsconfig.json`: `strict: true`, `allowJs: false`, `isolatedModules: true`
- Discriminated unions: `ActiveNote`, `ComposerAttachment`, `PostStatus`
- `as const` для query keys и enum-like массивов
- Repository interfaces с полной типизацией возвратов

**Проблемы:**
- `shared/types/index.ts:1-210` — monolithic types file. 210 строк, все domain types в одном месте. По мере роста — разбить по entity (post types → entities/post/model/types.ts)
- `httpClient.ts:50,54` — `undefined as T` — unsafe cast. TypeScript не может проверить это
- `composer-store.tsx:119,171` — хардкод строки `"сейчас"` вместо использования `Date` или i18n

### 2.2 Error Handling — ❌ Критично

**Нет глобального Error Boundary** — unhandled ошибки крашат приложение.

**Конкретные проблемы:**

1. **`window.alert()`** в `composer-store.tsx:83` — `window.alert("Активируйте LLM модель.")`. Это не модально, блокирует поток, выглядит unprofessionally. Нужен toast notification (sonner, react-hot-toast) или modal dialog.

2. **`window.confirm()`** в `useDeleteNote.ts:25` — `window.confirm("Удалить заметку...")`. Та же проблема. Нужен confirmation dialog из UI library.

3. **Fire-and-forget мутации** — `void createChat.mutateAsync(...)` в `composer-store.tsx:123,130,148,174,179,185` и `useDeleteNote.ts:27,33`. Ошибки мутаций молча проглатываются. Нет `onError` callback, нет пользовательского feedback.

4. **`console.error`** в `MswProvider.tsx` для MSW startup failures — нет пользовательского error state.

**Рекомендации:**
- Добавить `ErrorBoundary` в `AppProviders.tsx` как wrapper
- Заменить `window.alert/confirm` на toast + modal из shadcn
- Добавить `onError` handlers во все `void` мутации
- Создать `useErrorToast()` hook для единообразного error feedback

### 2.3 Компоненты — ✅ Хорошо

**Сильные стороны:**
- Screen components — thin wrappers, вся логика в hooks (`useFeedScreen`, `usePostScreen`)
- PascalCase файлы, barrel exports, чёткие имена
- shadcn/ui + CVA variant system для UI primitives

**Проблемы:**
- `FeedScreen.tsx:45` — template literal className: `` `feed-screen-wrap${layoutClassName}` ``. Это fragile — если `layoutClassName` пустой, получается `feed-screen-wrap`. Лучше `cn("feed-screen-wrap", layoutClassName)`
- Нет пропсовых типов для многих screen/widget components — props inline в функции. Рекомендуется выносить в type definitions

### 2.4 Хуки — ✅ Хорошо

**Сильные стороны:**
- Custom hooks везде где нужна логика (`useMediaQuery`, `useMobile760`, `useScreenBack`)
- Entity-level hooks с TanStack Query (`usePosts`, `useCreatePost`)
- `useCallback`/`useMemo` везде где нужно для стабильности ссылок

**Проблемы:**
- `usePost(id)` в `usePosts.ts:17-29` — делает `posts.list()` для каждого detail query. Это N+1 проблема при масштабировании. Нужен `posts.getById(id)` в repository interface
- `composer-store.tsx` — 225 строк в одном файле. Разбить на `composer-provider.tsx` + `composer-hooks.ts`

## 3. Масштабируемость

### 3.1 Позитивные сигналы

- FSD + boundaries предотвращают spaghetti code
- Repository pattern — замена бэкена (seed/msw/http) не требует изменений в UI
- TanStack Query + query key factory — кэширование и invalidation масштабируются
- Zustand stores изолированы по domain

### 3.2 Потенциальные проблемы

1. **Типы в `shared/types/index.ts`** — монолитный файл. При добавлении новых entities будет расти линейно. Рекомендация: colocate types с entities (`entities/post/model/types.ts`)

2. **`Post` type содержит `LocalNote[]` и `LocalChat[]`** (`shared/types/index.ts:92-104`). Post becomes a god object — вложенные массивы нот и чатов делают кэширование и обновление сложными. Рекомендация: нормализовать data model (отдельные queries для notes и chats по postId)

3. **MSW handlers (`handlers.ts`)** — in-memory store без persistence. При масштабировании тестов это может стать проблемой (concurrent test isolation). Seed repositories в `seedRepositories.ts` имеют ту же проблему.

4. **Нет виртуализации списков** — `FeedScreen`, `NotesScreen`, `ChatsScreen` рендерят все элементы. При 100+ постах/чатах это будет проблемой. Добавить `@tanstack/react-virtual` для длинных списков.

5. **Static export** (`output: "export"`) — ограничивает функциональность Next.js (нет SSR, server actions, middleware). Это осознанный выбор для GitHub Pages, но ограничивает future features.

### 3.3 Производительность

- **Memoization**: обильное использование `useMemo`/`useCallback` (292 совпадения в коде). Это хорошо для React 19, но стоит убедиться что не происходит over-memoization
- **Query config**: `staleTime: 60s`, `gcTime: 5min`, `retry: 1` — разумные значения
- **Нет lazy loading** компонентовeyond MSW. `next/dynamic` для тяжёлых виджетов (charts, analytics) улучшил бы initial load
- **CSS**: hybrid Tailwind + legacy CSS + tokens — работает, но 17 CSS файлов в `app/styles/` — потенциальный source of conflicts при росте

## 4. Maintainability

### 4.1 Позитивные сигналы

- **Документация**: `docs/` с product specs, UX flows, engineering docs — excellent for onboarding
- **Test coverage**: Vitest unit + Playwright E2E. `npm run check` = typecheck + lint + test + build
- **ESLint boundaries**: автоматическое предотвращение архитектурных нарушений
- **Glossary** (`docs/glossary.md`) — документирует domain terminology

### 4.2 Проблемы

1. **Нет git hooks** — нет pre-commit lint/test. Команды полагаются на `npm run check` вручную. Рекомендация: добавить `husky` + `lint-staged`

2. **Нет CI/CD** — нет GitHub Actions. Даже для personal project рекомендуется basic CI (typecheck + lint + test on PR)

3. **Нет i18n** — хардкод русского языка в коде (`"Загрузка ленты…"`, `"Активируйте LLM модель."`, `"сейчас"`). Если проект будет multi-language — потребуется рефакторинг. Если нет — это OK.

4. **No env validation library** — `dataSource.ts` делает ручную валидацию. Для проекта с 3 env vars это acceptable, но `@t3-oss/env-nextjs` дал бы type safety + runtime validation.

5. **`composer-store.tsx`** — 225 строк, 20+ dependencies в `useCallback`. Этот файл трудно тестировать и поддерживать. Рекомендация: разбить на smaller units.

### 4.3 Testing

**Сильные стороны:**
- Co-located tests (`*.test.ts` рядом с source)
- 3 уровня: unit (Vitest), integration (Vitest + RTL), E2E (Playwright)
- MSW для реалистичного мока API

**Рекомендации:**
- Нет tests для screens/widgets — только shared lib + entities. Рекомендуется добавить smoke tests для ключевых screens
- `useDeleteNote.ts` — тестировать user interaction flow (confirm → delete → navigation)
- `composer-store.tsx` — нет unit tests. Этот complex provider должен иметь тесты
