# Качество и CI/CD

Раздел о том, как я обеспечиваю качество кода и стабильность релизов: от локальных проверок до продакшн-мониторинга.

Цель: чтобы релизы были предсказуемыми, а проблемы ловились как можно раньше и дешевле.

## Принципы качества

Базовые принципы:

- quality by default: проверка встроена в процесс, а не "в конце как отдельный этап";
- automation first: повторяемые проверки автоматизируются;
- risk-based testing: глубина проверки зависит от стоимости ошибки;
- release with observability: релиз без мониторинга не считается завершенным.

## Что входит в контур качества

Минимальный рабочий набор:

- статические проверки (линтеры, форматтеры, type checks);
- unit/integration/e2e тесты для критичных сценариев;
- code review для значимых изменений;
- проверка миграций и контрактов интеграций;
- post-release мониторинг и triage инцидентов.

## Code review

В ревью проверяю:

- корректность бизнес-логики;
- понятность и поддерживаемость кода;
- надежность обработки ошибок;
- риск регрессии в затронутых сценариях;
- безопасность и права доступа в чувствительных местах.

Ревью - это не "поиск стиля", а фильтр рисков перед релизом.

## Автоматические проверки в CI

Обычно в pipeline включены:

- install/build;
- линтинг и проверки типов;
- запуск тестов;
- проверка сборки артефактов;
- базовые security checks (зависимости/секреты, где применимо).

Логика простая: если обязательный gate не прошел, релиз не идет дальше.

## Стратегия тестирования

Тестовый контур строится по критичности:

- unit: локальная бизнес-логика и утилиты;
- integration: взаимодействие сервисов и данных;
- e2e: ключевые пользовательские сценарии;
- regression: набор кейсов, которые уже ломались или были рисковыми.

Для AI-функциональности отдельно добавляю eval-подход и сценарные проверки поведения.

## CI/CD и релизная дисциплина

Релизный процесс обычно включает:

- подготовку изменения в изолированной ветке;
- автоматические проверки в CI;
- staged deployment (где уместно);
- rollback-стратегию на случай деградации;
- контроль состояния после выкатки.

Цель - чтобы релиз можно было повторить и откатить без ручной магии.

## Мониторинг и алертинг в продакшне

После релиза отслеживаю:

- error rate и критичные исключения;
- latency и деградации производительности;
- доступность ключевых интеграций;
- бизнес-метрики критичных сценариев.

Алерты должны быть настроены на реальные инциденты, а не на шум.

## Работа с инцидентами и регрессиями

Алгоритм:

1. быстрое обнаружение и локализация;
2. восстановление сервиса;
3. root-cause анализ;
4. фикс + regression test;
5. обновление runbook/документации.

Так проблема становится закрытым циклом, а не повторяющимся "пожаром".

## Как оцениваю качество релиза

Релиз считаю хорошим, если:

- критичные сценарии стабильны;
- риск регрессий контролируется тестами;
- мониторинг показывает нормальную эксплуатацию;
- команда понимает, что и как улучшать в следующей итерации.

## Анти-паттерны

- ручные релизы без проверок и повторяемого pipeline;
- "тесты потом, сейчас быстрее выпустим";
- отсутствие rollback-плана;
- отсутствие мониторинга после выката;
- исправление инцидента без добавления regression-кейса.

## Практический checklist

Перед релизом проверяю:

- прошли линтеры, type checks и тесты;
- выполнен code review критичных изменений;
- подтверждена корректность миграций/контрактов;
- есть план отката;
- включен мониторинг и настроены алерты;
- определен owner на post-release период.
